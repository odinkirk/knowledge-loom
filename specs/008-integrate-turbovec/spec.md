# Feature Specification: Integrate turbovec

**Feature Branch**: `008-integrate-turbovec`  
**Created**: 2026-05-25  
**Status**: Draft  
**Input**: User description: "We will integrate turbovec"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Vector Search with Turbovec (Priority: P1)

A user opens their Obsidian vault in knowledge-loom, indexes it, and performs a semantic search. The system returns semantically similar notes using turbovec's approximate nearest neighbor (ANN) search instead of the previous brute-force cosine-distance scan. Results include file paths and relevance scores.

**Why this priority**: This is the core value proposition — faster, more memory-efficient semantic search. Without this, turbovec adds no user-facing benefit.

**Independent Test**: Index a test vault of 1000 markdown files, run a semantic query ("machine learning"), and verify that results are returned within acceptable latency with similarity scores comparable to the previous backend.

**Acceptance Scenarios**:

1. **Given** a vault has been fully indexed, **When** the user searches for "quantum physics", **Then** the system returns notes sorted by semantic relevance with similarity scores
2. **Given** an empty vault (no indexed files), **When** the user searches, **Then** the system returns an empty result set without errors
3. **Given** a vault with 10,000 indexed chunks, **When** the user searches, **Then** results are returned faster than or equal to the previous sqlite-vec backend

---

### User Story 2 - Vault Indexing into Turbovec (Priority: P1)

A user indexes their vault (via `loom_reindex` or daemon mode file watching). Each markdown file is chunked, embeddings are generated by the configured embedding provider, and the vectors are stored in a turbovec index on disk. Files can be added, updated (re-embedded), and removed from the index.

**Why this priority**: Search is useless without indexing. This is the other half of the core loop.

**Independent Test**: Run a full vault reindex, verify the turbovec index file is created on disk, confirm the number of indexed vectors matches the expected chunk count, then delete a file and verify its vectors are removed.

**Acceptance Scenarios**:

1. **Given** a vault with markdown files, **When** the user triggers a full reindex, **Then** all file chunks are embedded and stored in the turbovec index
2. **Given** an existing indexed vault, **When** a file is modified (in daemon mode), **Then** its old embeddings are removed and new ones are inserted
3. **Given** an existing indexed vault, **When** a file is deleted, **Then** its embeddings are removed from the index
4. **Given** a vault is being indexed, **When** one file fails to embed (e.g., embedding provider error), **Then** that file is skipped gracefully and indexing continues for remaining files

---

### User Story 3 - Graph-Aware Filtered Search (Priority: P2)

A user searches for "blockchain" and the system prioritizes results from notes that are graph-connected to the user's currently open note. The turbovec allowlist mechanism restricts the ANN search to a subset of candidates derived from graph neighbors, improving result relevance without scanning the entire index.

**Why this priority**: This is a differentiated capability unlocked by turbovec's filtered search — it enhances the existing graph-fused search with proper sub-index scoping rather than post-hoc re-ranking.

**Independent Test**: Open a note about "Ethereum," search for "smart contracts," and verify that graph-neighbor notes appear higher in results than they would in an unfiltered search.

**Acceptance Scenarios**:

1. **Given** a note with 5 graph-linked neighbors, **When** the user performs a graph-aware semantic search, **Then** the allowlist constrains search to only those neighbor candidates (plus a fallback set)
2. **Given** a note with no links (orphan note), **When** the user performs a graph-aware search, **Then** the system falls back to an unfiltered full-index search

---

### User Story 4 - Index Persistence Across Restarts (Priority: P2)

A user indexes their vault, closes knowledge-loom, and reopens it later. The turbovec index is loaded from disk without requiring a full reindex, making startup fast even for large vaults.

**Why this priority**: Without persistence, every restart would require a costly full reindex. This is essential for practical daily use.

**Independent Test**: Index a vault, restart the server, and verify that semantic search works immediately without reindexing.

**Acceptance Scenarios**:

1. **Given** a previously indexed vault, **When** knowledge-loom starts, **Then** the turbovec index is loaded from disk
2. **Given** a corrupt or missing index file on disk, **When** knowledge-loom starts, **Then** the system falls back to a fresh empty index and logs a warning

---

### User Story 5 - Memory Efficiency for Large Vaults (Priority: P3)

A user with a large vault (50,000+ notes) can search with acceptable memory usage. The turbovec quantization compresses embedding vectors so the index fits in a fraction of the memory that raw float32 vectors would require.

**Why this priority**: This is a quality-of-life improvement for power users. Smaller vaults won't notice the difference, but large vaults benefit significantly.

**Independent Test**: Index a vault with 50,000 chunks of 384-dim embeddings, and verify the in-memory index size is no more than ~25% of the equivalent raw float32 storage.

**Acceptance Scenarios**:

1. **Given** a vault with 50,000 indexed chunks, **When** the index is loaded into memory, **Then** memory usage is substantially lower than storing raw float32 vectors
2. **Given** a vault with embeddings from different providers (varying dimensions), **When** vectors are added to the index, **Then** the index correctly handles the configured dimension size

---

### Edge Cases

- What happens when the embedding provider returns vectors of a different dimension than the existing index? (e.g., switching from local 384-dim to Ollama 768-dim)
- How does the system handle a turbovec index file that was created with a different quantization level?
- What happens when the index file is partially written (e.g., process crash during save)?
- How does incremental reindex work when only some files have changed — can individual vectors be updated in-place?
- What happens when a search is requested before the index is fully loaded from disk?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST replace the sqlite-vec vector store with turbovec as the ANN vector search backend
- **FR-002**: System MUST store embedding vectors in a turbovec index with configurable quantization (4-bit default)
- **FR-003**: System MUST persist the turbovec index to disk so it survives server restarts
- **FR-004**: System MUST support adding, updating, and removing individual vectors from the index without a full rebuild
- **FR-005**: System MUST support filtered ANN search where search is constrained to a subset of vectors via an allowlist of IDs
- **FR-006**: System MUST return similarity scores alongside matched paths in search results
- **FR-007**: System MUST continue to support all three existing embedding providers (local fastembed, Ollama, OpenRouter) with no changes to their interfaces
- **FR-008**: System MUST gracefully handle index file corruption by logging a warning and initializing a fresh index
- **FR-009**: System MUST map internal note/chunk identifiers to turbovec's uint64 ID space for both storage and allowlist filtering
- **FR-010**: System MUST handle dimension mismatches between the stored index and new embeddings by reporting a clear error and skipping the affected files
- **FR-011**: System MUST automatically migrate existing sqlite-vec embeddings into the turbovec index on first startup after upgrade, then remove the old embeddings database
- **FR-012**: System MUST support concurrent search and index mutation without data corruption; concurrent searches see a consistent snapshot of the index

### Key Entities

- **Vector Index Entry**: Represents a single embedded chunk — associates a unique uint64 ID with a float vector and metadata (file path, heading, content). Stored in turbovec's compressed index structure.
- **Index Metadata**: Configuration persisted alongside the vector index — records the quantization level, embedding dimension, and index version for validation on reload.
- **ID Map**: A bidirectional mapping between knowledge-loom's internal identifiers (file path + chunk ordinal) and turbovec's uint64 vector IDs, enabling both direct lookup and allowlist-based filtered search.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Semantic search latency for a 10,000-chunk vault is ≤100ms at p95 (the existing Constitution VII target for vector search, which the previous sqlite-vec backend met)
- **SC-002**: Memory usage for a 50,000-chunk index (384-dim embeddings) is reduced by at least 50% compared to storing raw float32 vectors
- **SC-003**: Index load time from disk for a 10,000-chunk vault is under 2 seconds
- **SC-004**: All existing search-related tests continue to pass without modification to their expected behavior (same inputs produce equivalent result ordering)
- **SC-005**: A full vault reindex of 1,000 files completes without errors and produces a usable index for immediate search
- **SC-006**: The system handles the removal and addition of individual file embeddings within a running session without requiring a full index rebuild
- **SC-007**: Recall@10 (fraction of top-10 exact-search results also found by turbovec) is >= 0.95 on a standard test corpus of >= 1,000 embedded chunks

## Assumptions

- The turbovec Rust crate (`turbovec`) is published and stable enough for a dependency — if the API is unstable, it will be pinned to a specific version
- Users have either ARM (NEON) or x86 (AVX2/AVX-512BW) CPUs — turbovec's SIMD kernels cover these architectures
- The existing embedding provider interfaces (`EmbedProvider` trait) remain unchanged
- The existing chunking pipeline (`src/chunks.rs`) and markdown parsing remain unchanged
- The existing RRF search merging logic in `src/search.rs` is preserved; only the vector search backend is swapped
- The sqlite-vec dependency (`src/index.rs`) is removed entirely after migration; it is not kept as a parallel fallback
- On first startup after upgrade, existing sqlite-vec embeddings are automatically migrated into the turbovec index, then the old `embeddings.db` is deleted
- Quantization defaults to 4-bit as a balance between memory savings and search accuracy
- Index IDs are deterministic across reindex runs for the same chunk (path + heading), enabling stable allowlist references

## Clarifications

### Session 2026-05-25

- Q: How should the system handle the transition from sqlite-vec to turbovec for existing users? → A: One-time automatic migration on first startup — read embeddings from sqlite-vec and write them into turbovec, then delete embeddings.db
- Q: Should there be a formal accuracy criterion for turbovec vs exact float32 search? → A: Yes — recall@10 >= 0.95 versus exact float32 search on a standard test corpus
- Q: What should happen when a search and an index mutation overlap? → A: Search and mutation must be safe to call concurrently — search sees a consistent snapshot of the index
